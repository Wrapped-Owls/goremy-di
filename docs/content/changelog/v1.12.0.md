### Added

- **`SingleStorage`**: Zero-allocation storage backend optimized for sub-injectors that hold exactly one dependency. The
  entry is kept in flat struct fields, avoiding any heap allocation beyond the struct itself.

- **`SliceStorage`**: Flat-slice storage backend for a small, known number of entries (2–4). A single pre-allocated
  contiguous slice is used, keeping every lookup in one cache line and avoiding map overhead entirely.

- **`NewStorage` dispatcher**: `stgbind.NewStorage(opts, size)` selects the most efficient backend at construction time:

| `size` | Backend                      |
|--------|------------------------------|
| 1      | `SingleStorage`              |
| 2 – 4  | `SliceStorage`               |
| > 4    | `ElementsStorage` (hash map) |

- **`NewWithStorage` on `StdInjector`**: New constructor that accepts an explicit storage backend, allowing callers to
  supply a pre-sized lightweight storage instead of always allocating the default map-backed `ElementsStorage`.

- **Storage benchmarks**: `BenchmarkStorage_Set` and `BenchmarkStorage_Get` were added to `remy/internal/stgbind`. Both
  are table-driven via `NewStorage`, covering all size buckets in a single run for easy cross-implementation comparison.

### Changed

- **`GetWithPairs`** now calls `stgbind.NewStorage(injopts.CacheOptNone, len(elements))` instead of always allocating a
  map, automatically selecting the optimal backend based on the number of pairs provided.
- **`GetWith`** now uses `stgbind.NewSliceStorage` (capacity 4) instead of the map-backed storage, matching the typical
  small-override usage pattern.

### Fixed

- **Nested dependency error propagation in duck-typed interface resolution**: `Get` now preserves the real nested
  `ErrElementNotRegistered` when interface duck typing reaches a candidate bind whose own dependencies are missing. This
  fixes cases where the library incorrectly reported the requested interface as unregistered instead of the actual
  missing dependency such as `context.Context`.

### Performance

Benchmarks show that `SingleStorage` is **~5× faster** than `ElementsStorage` on write and **zero-allocation** on read
for a single entry. `SliceStorage` achieves **~2–3× faster** writes than the map for 2–4 entries while also producing
zero allocations on reads. The map-backed `ElementsStorage` remains the right choice for five or more entries, where its
O(1) lookup amortizes the higher construction cost.
